## Контекст и проблема

Живой прогон Storm на реальной задаче дал ценный результат и вскрыл изъяны.

**Сработало (закрепляем как инварианты, не ломаем):**
- `[FINDING]` run/expects/observed с компилируемыми пруфами — движки давали
  исполняемые скрипты с verdict; владелец мог перепроверить, а не верить на
  слово (verify-don't-trust в чистом виде).
- Дивергенция > консенсус — каждый движок нашёл *разный* реальный баг; поодиночке
  владелец взял бы один и пропустил остальные. Подтверждает Mixture-of-Agents.

**Провалы:**
1. **Один движок вешает весь совет.** glm завис на 30 минут (CPU 0.1%, без
   вывода), а `fan-out` ждёт всех (`Promise.allSettled`) → совет не завершился.
   Это прямой tradeoff решения no-timeouts («задедлокшийся-но-живой процесс висит
   до Ctrl-C»), который укусил вживую.
2. **Ноль видимости прогона.** companion отдаёт один JSON в самом конце; 30 минут
   не видно, жив ли кто-нибудь.
3. **Галлюцинации о моделях.** Вызывающая сессия описала запущенные движки как
   «gemini-3.1-pro / gpt-5.4 / sonnet-4.6», хотя реально шли claude(opus) / codex
   (gpt-5.5) / glm(glm-5.2). Причина: модель не видна в argv (codex берёт её из
   `~/.codex/config.toml`, claude брал дефолт CLI) → сессия достроила из памяти.
4. **Молчаливый выпад движка.** gemini вернул `no_result` (507 байт) без видимой
   причины.

**Ключевая переформулировка владельца:** нужен **детектор отсутствия прогресса**,
а не лимит времени. Движок, работающий долго, но шлющий прогресс — это норма
(ему столько нужно). Движок живой, но переставший делать прогресс — это **баг**,
и только его режем. Liveness = «движок *делает* прогресс», не «процесс жив» и не
«время вышло».

## Эмпирика: сигнатура прогресса каждого движка

Два инструментированных спайка через реальные адаптеры Storm. Метрика —
**самая длинная пауза тишины при нормальной работе** (выше неё = зависание).

**Спайк 1 — read-only аналитическая задача (`proof:false`):**

| Движок | Время | Чанков | Макс. пауза |
|---|---|---|---|
| claude (opus) | 263с | 447 | 3.9с |
| glm (glm-5.2) | 501с | 7480 | 14.9с |
| codex (gpt-5.5/xhigh) | 159с | 126 | **66.9с** |

codex молчит до ~67с во время *чистого финального reasoning* (нет промежуточных
tool-границ → единственный длинный провал).

**Спайк 2 — proof-режим (worktree + полные права + запись файла + прогон),
с файловым монитором:**

| Движок | Время | Stream-чанков | File-событий | Макс. пауза | Файлов *в* паузах |
|---|---|---|---|---|---|
| claude | 153с | 272 | 2 | 4.1с | — |
| glm | 62с | 1207 | 2 | 10.1с | 0 |
| codex | 44с | 50 | 1 | 7.3с | — |

**Выводы:**
- У всех трёх есть детектируемый сигнал прогресса. Чёрного ящика нет; codex просто
  грубее (item-границы в stderr), и его худшая пауза конечна и измерима.
- **Файловая активность избыточна.** Запись файла совпадает со stream-событием
  (tool-call эмитится в тот же миг); во *все* паузы stdout попало 0 файловых
  изменений (`filesInGap=0`). Отдельный fs-мониторинг не нужен (YAGNI). Stream
  самодостаточен.
- Action-задачи держат codex плотным (7.3с против 66.9с) — длинный провал бывает
  только на чистом reasoning. Порог обязан покрыть худший случай.

**Пороги зависания (макс. пауза из обоих спайков + запас):**

| Движок | Макс. норма | Порог `stallMs` | Запас |
|---|---|---|---|
| claude | 4.1с | **20000** | 5× |
| glm | 14.9с | **60000** | 4× |
| codex | 66.9с | **180000** | 2.7× |

Числа на фактах. codex получает меньший кратный запас (грубее сигнал), но 180с
против нынешнего «вечно» — радикально; настраивается в config.

## Блок A — детект зависания + partial synthesis

Механизм уже есть в `run-engine.mjs` (`lastActivity` + `armStall`); его выключили
глобально (`stallMs:null`) из-за codex. Включаем **per-engine** и доводим.

**A1. Per-engine порог в `config.json`:**
```json
"engines": [
  { "id": "claude", "model": "opus",                       "stallMs": 20000  },
  { "id": "codex",                                          "stallMs": 180000 },
  { "id": "glm", "model": "glm-5.2", "effort": "max",       "stallMs": 60000  }
]
```
Глобальный `config.stallMs` остаётся как fallback. Отсутствие `stallMs` у движка →
наследует глобальный (=`null` сейчас) → stall выключен (backward-compat).

**A2. Проводка per-engine.** Сейчас `fan-out` раздаёт один `opts.stallMs` всем.
Меняем: для каждого движка берётся `e.stallMs` → `runEngine(..., { stallMs: e.stallMs })`.
Каждый армится своим порогом.

**A3. Сигнал прогресса (уже верный).** `onActivity` обновляет `lastActivity` на
любой байт stdout/stderr; `armStall` сбрасывает таймер. Под stream-флагами
claude/glm шлют NDJSON непрерывно, codex — item-события в stderr. Тишина дольше
`stallMs` **при живом процессе** → SIGKILL → `status:'stalled'`. Это детект
отсутствия прогресса, не wall-clock: общее время не ограничено.

**A4. Partial synthesis.** `fan-out` уже на `Promise.allSettled`. Зарезанный по
stall движок **резолвится** (`stalled`), не висит → совет завершается, когда все
резолвлены: работающих ждём сколько нужно (шлют прогресс), зависших режем по их
порогу. Worktree зарезанного чистится (`ws.cleanup` в `finally` — уже есть).
Синтез идёт из `ok`/`salvaged`; выпавшие перечислены с причиной (Блок B).

Инвариант: ни один движок не висит дольше своего `stallMs`; реально работающий
не режется никогда.

## Блок B — прозрачность прогона

**B1. Heartbeat прогресса в stderr.** `run-engine` принимает `opts.onProgress({chunks,
lastActivityMs})`, вызывается на каждом `onActivity` (дёшево). `fan-out` собирает
состояние движков и раз в ~15с пишет сводку в **stderr** (stdout чист под финальный
JSON — оркестратор его парсит):
```
[storm +45s] claude: 130ev idle 2s | codex: 38ev idle 5s | glm: 410ev idle 1s
[storm +60s] claude: 198ev idle 1s | codex: 41ev idle 18s | glm: stalled@60s
```

**B2. Resolved-model echo.** `run-engine` захватывает **фактическую** модель из
вывода движка:
- claude/glm: stream-json событие `system/init` → `ev.model`
- codex: regex `model:\s*(.+)` по session-заголовку в stderr (+ `reasoning effort`)
- gemini: из `cfg.model`

→ `result.resolvedModel`, companion отдаёт в JSON. Синтез показывает:
`claude: claude-opus-4-8`, `codex: gpt-5.5 (xhigh)`, `glm: glm-5.2`. Это реально
запущенное, из вывода движка — галлюцинации о моделях исключены by construction.

**B3. no_result-reason в синтез.** Причина уже в коде (`run-engine` кладёт
`no_marker`/`unterminated`/`empty` + хвост stderr в поле `error`); companion её
отдаёт. Добавляем **контракт синтеза** в `commands/storm.md` + `SKILL.md`: оркестратор
обязан перечислить каждый движок со `status != ok` и его причиной — не синтезировать
молча из подмножества. (Детальный лог *почему* у gemini — turns exhausted / пустой
ответ — отложен вместе с возвратом gemini, он сейчас вне пула.)

## Что НЕ входит / инварианты

- Верхний wall-clock cap на «живой, но бесконечно стримящий» движок (шлёт события,
  но не заканчивает) — редкий эджкейс, YAGNI; добавим опциональный backstop, если
  всплывёт.
- Возврат gemini/antigravity в пул и детальный gemini-reason — отдельный заход.
- Не трогаем: `[FINDING]` контракт (`prompt.mjs`); proof-режим self-experiment +
  verify-don't-trust (`annotateWithProof`, worktree); дивергенция > консенсус (B3
  усиливает перечисление, не схлопывает находки); zero-dep / ESM / node:test;
  cwd/secrets-изоляция.
- Backward-compat: правки независимы от `proof.enabled`; конфиг без `stallMs` =
  поведение как сейчас.

## Тестирование

node:test, zero-dep. `fan-out` тестируется через `opts.runner` (фейковые движки),
`run-engine` — через фейковый child stream. **Все таймеры управляемые**
(контролируемая эмиссия чанков), никаких реальных `sleep` — прошлый flaky
timing-тест не повторяем.

Ключевые тесты:
- **Защита от рецидива:** движок, шлющий событие каждые `stallMs/2` в течение
  `3×stallMs`, **не** режется (armStall сбрасывается) — «работает долго, но шлёт
  прогресс = не трогаем».
- Контраст: молчащий дольше `stallMs` → `stalled`, promise резолвится (не висит).
- Per-engine: разные движки с разными `stallMs` армятся каждый своим.
- Partial synthesis: один `stalled` + два `ok` → возвращаются все три, `stalled`
  помечен, без зависания.
- resolved-model: парсинг `system/init` (claude/glm), codex-header regex,
  gemini-config — тест на источник.
- heartbeat: `onProgress` вызывается; формат сводки.

## Пересмотр no-timeouts (обоснование)

Решение 2026-06-25 убрало таймауты, потому что claude/glm молчали во время
reasoning, а codex под xhigh молчит до ~60с — слепой stall ложно убивал
работающих. Тогда heartbeat ещё не было.

Теперь heartbeat есть (stream-json для claude/glm; item-границы в stderr для
codex), и эмпирика дала максимальные нормальные паузы. Поэтому per-engine stall
с откалиброванными порогами режет **только** реально молчащего, а работающий
(шлёт события) не трогается. Принцип «liveness, не часы» сохранён — уточнён:
liveness = прогресс потока, измеренный per-engine, а не отсутствие любого
ограничения. Tradeoff «зависший висит вечно» закрыт.
