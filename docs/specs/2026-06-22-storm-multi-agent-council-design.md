# Storm — мульти-агентный совет для Claude Code (дизайн)

- Дата: 2026-06-22
- Статус: дизайн одобрен, спека на ревью
- Репозиторий: личный (отдельный от whytap), плагин Claude Code
- Лицензия: Apache-2.0 (структура заимствована у `codex`-плагина OpenAI, с атрибуцией)

## Контекст

При обычной работе один агент не всегда выбивает максимальное качество с первого
прохода и не сразу находит баги — это стоит итераций и времени. Эмпирически
(эксперимент «Storm»: одна задача параллельно в Gemini + OpenAI + Claude, у
каждого свои explorer-агенты и поиск по репозиторию) **тройка независимых
фронтир-моделей в слепом сравнении всегда побеждала любую одиночную модель**.
Причина — разные веса/данные/архитектуры дают некоррелированные ошибки, а
синтез собирает правильные куски (см. Mixture-of-Agents).

Цель: воспроизвести это как инструмент. Claude остаётся кор-агентом в живой
беседе; по явной команде подключается совет из трёх движков; Claude синтезирует
единый ответ. Это **усиление по запросу**, не на каждый вопрос.

## Решение

Плагин Claude Code с командой `/storm`. Два режима:

- **`/storm plan <задача>`** (v1) — read-only усиление: три движка как
  эксплореры/ревьюеры (RCA, поиск причины, ревью, анализ). Каждый возвращает
  компактный результат, Claude синтезирует. Ничего не пишется.
- **`/storm action <задача>`** (фаза 2) — параллельная имплементация: три git
  worktree, каждый движок пишет код в своём; Claude ревьюит три diff и собирает
  результат умным мёржем.

v1 = только `plan`. Тестируемся на нём (ревью тройкой самоценно), затем `action`.

## Движки (адаптеры)

Единый интерфейс адаптера: вход `(задача, роль, контекст)` → выход
`(нормализованный STORM_RESULT)`. Список движков — в конфиге; добавление движка =
новый адаптер + строка конфига, без переделки ядра.

| Голос | Вызов | Модель | Заметки |
|---|---|---|---|
| Claude | независимый субагент Claude Code (фон), **не** главный тред | модель сессии (сейчас Opus 4.8) | отдельный голос, не дубль оркестратора; модель не из `agy`, а из харнесса |
| Codex | `codex` headless (фон) | из `~/.codex/config.toml` (gpt-5.x) | флаги модели/effort не форсим |
| Antigravity | `agy --model "Gemini 3.1 Pro (High)" -p … --dangerously-skip-permissions --print-timeout` | Gemini 3.1 Pro (High) | гео-обход через Antigravity-бэкенд |

Подтверждено действием (2026-06-22):
- `agy` 1.0.10, headless `-p` работает, `--print-timeout` (default 5m) — защита
  от non-TTY hang.
- `--model "Gemini 3.1 Pro (High)"` принимается дословно, baseline ~9с на
  простом промпте.
- `agy models`: Gemini 3.5 Flash (Low/Med/High), Gemini 3.1 Pro (Low/High),
  Claude Sonnet/Opus 4.6, GPT-OSS 120B. Reasoning-effort зашит в имя модели.
- Дефолт модели не хранится в `~/.gemini/config/config.json` (серверный) →
  для детерминизма модель **форсим флагом**.

Через `agy` доступны и Claude, и GPT-OSS, но для разнообразия мозгов берём
**только Gemini** (Claude через agy = дубль оркестратора; GPT-OSS 120B ≠ Codex
gpt-5.x, Codex берём напрямую как более сильный).

## Поток данных (plan v1)

```
/storm plan <задача>
  → оркестратор (главный Claude) формулирует подзадачу + роль под контекст
  → ПАРАЛЛЕЛЬНЫЙ fan-out:
       • Claude-субагент (Agent, фон)
       • Codex (Bash, фон)
       • agy Gemini 3.1 Pro High (Bash, фон)
  → ждём всех (время ≈ самого медленного, не сумма)
  → каждый адаптер извлекает <STORM_RESULT>, режет сырьё
  → СИНТЕЗ (Claude):
       ✓ консенсус всех троих   = high-confidence
       ⚠ расхождения            = флаг пользователю на ревью
       + уникальные находки каждого
  → один ответ в живой беседе
```

## Инвариант: защита контекста

Мой (главного треда) контекст **никогда не видит сырой stdout движка**. Между
ними — адаптер. Прецедент проблемы: Codex отдавал «левые сообщения» (прогресс,
рассуждения), раздувая контекст.

- **Контракт вывода.** Промпт каждому движку требует вернуть результат только
  внутри маркеров, без рассуждений/прогресса:
  ```
  <STORM_RESULT>
  …компактный структурированный результат (bullets / схема)…
  </STORM_RESULT>
  ```
- **Адаптер извлекает только блок между маркерами.** Прогресс/болтовня
  отбрасываются до попадания в контекст оркестратора.
- **Парсим вывод, не exit-код** (CLI дают exit 0 при ошибке — проверено на
  gemini `UNSUPPORTED_LOCATION`).
- **Codex-специфика** (известная боль): извлечение последнего assistant-message
  + маркер; точный механизм для `codex` headless тестируется при сборке
  адаптера — заложено явным пунктом плана, не «потом».
- **В `action` раздувания нет by design**: выхлоп движка — код в worktree,
  оркестратор смотрит `git diff`, не stdout.

## Обработка ошибок / надёжность

- Таймаут на каждый движок (`--print-timeout` у agy; обёртка-таймаут у
  остальных).
- Деградация: движок упал/таймаут → Storm не падает, синтез из ответивших +
  явная пометка «<движок> не ответил».
- Rate-limit: тройка только по команде → расход в норме.
- Латентность ≈ медленный движок + синтез; не для мелочей (для них — обычный
  диалог).

## Упаковка (плагин-репо)

```
storm/
  .claude-plugin/plugin.json     # манифест
  commands/storm.md              # /storm
  skills/storm-runtime/          # контракт вызова (à la codex-cli-runtime)
  skills/engine-adapters/        # claude | codex | antigravity
  scripts/storm-companion.mjs    # мульти-движковый runtime: fan-out → сбор
  LICENSE (Apache-2.0) + NOTICE  # атрибуция codex-плагина
  README.md
```

Структуру перенимаем у `codex`-плагина (companion-runtime + forwarder-agent +
skills/commands/hooks). Runtime пишем свой (мульти-движок). **v1 = ядро под нашу
тройку**; публичная универсальность (degradation без чужих движков, онбординг
чужих CLI, кроссплатформа, docs) — отдельная фаза.

## Тестирование

- Юнит: адаптеры — парсинг `<STORM_RESULT>`, обработка таймаута/ошибки, отсечение
  сырья.
- Контракт: `storm-companion` fan-out → сбор на мок-движках.
- Smoke: реальный `/storm plan` на тестовой задаче с живой тройкой.

## Action-режим (фаза 2, задизайнен)

- `storm-companion` создаёт 3 git worktree (по движку, своя ветка).
- Каждый движок пишет код в своём worktree (изоляция: Claude — субагент с
  worktree-isolation, Codex — `--write` в своём cwd, agy — в своём).
- Оркестратор ревьюит три diff и собирает **умным мёржем**:
  - **база = одна лучшая целостная реализация**;
  - из двух других — **точечный cherry-pick** изолированных улучшений (краевые
    случаи, удачный тест, именование), где не рвётся связность;
  - **обязательный прогон тестов** после сборки (гибрид никем не проверен
    целиком);
  - варианты разошлись концептуально → берём лучший целиком, остальные как
    ревью-замечания.
- Механическое сшивание кусков запрещено (даёт франкенштейна).
- Неиспользуемые worktree подчищаются.

## Решения

| Вопрос | Решение |
|---|---|
| Триггер | явная команда `/storm` (не авто, не на каждый вопрос) |
| Состав | тройка Claude + Codex + Antigravity (подтверждена действием) |
| Antigravity-модель | Gemini 3.1 Pro (High) |
| Консенсус | Claude синтезирует (консенсус / расхождения / уникальное) |
| Роль | гибкая под задачу (эксплорер / ревьюер / имплементер) |
| Режимы | `plan` (v1) и `action` (фаза 2) |
| Action-сборка | умный мёрж (база + cherry-pick + тесты), не выбор одного |
| Упаковка | плагин Claude Code, отдельный личный репо, Apache-2.0 |
| Защита контекста | маркеры `<STORM_RESULT>` + адаптер режет сырьё |
